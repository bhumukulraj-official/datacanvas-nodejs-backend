# Portfolio Website Backend Deployment Documentation

This document provides comprehensive instructions for deploying the portfolio website backend to various environments, including development, staging, and production.

## Deployment Environments

### 1. Development Environment

- **Purpose:** Local development and testing
- **Audience:** Developers
- **Infrastructure:** Local machines or development containers
- **Constraints:** Simulate production as closely as possible while optimizing for development speed

### 2. Staging Environment

- **Purpose:** Pre-production testing and QA
- **Audience:** QA team, product owners, developers
- **Infrastructure:** Cloud infrastructure similar to production but with reduced resources
- **Constraints:** Must mirror production configuration with minimal differences

### 3. Production Environment

- **Purpose:** Live user-facing system
- **Audience:** End users
- **Infrastructure:** High-availability cloud deployment
- **Constraints:** Performance, security, and reliability requirements

## Monolithic Deployment Architecture

```
┌─────────────────────┐     ┌─────────────────────┐
│    Load Balancer    │────▶│ Application Server  │
└─────────────────────┘     │  (Monolithic with   │
         ▲                  │  modular structure) │
         │                  └──────────┬──────────┘
┌────────┴────────┐                    │
│   CDN (Static)  │                    │
└─────────────────┘                    │
                                       │
                 ┌─────────────────────┼─────────────────────┐
                 │                     │                     │
                 ▼                     ▼                     ▼
      ┌─────────────────────┐ ┌─────────────────────┐ ┌─────────────────────┐
      │  PostgreSQL DB      │ │    Redis Cache      │ │ Object Storage      │
      └─────────────────────┘ └─────────────────────┘ └─────────────────────┘
```

The architecture follows the modular monolith approach outlined in the architecture document, with clear boundaries between modules to facilitate future microservices extraction as needed.

### Key Architecture Components

1. **Application Server**: Single Node.js/Express application with modular structure
2. **Database**: PostgreSQL instance with proper schema design
3. **Cache Layer**: Redis for caching and session management
4. **Object Storage**: For media files and static assets
5. **Load Balancer**: For distribution of traffic to application instances
6. **CDN**: For static assets delivery

## Deployment Strategy

The deployment strategy is aligned with the monolithic architecture design, while maintaining flexibility for future scaling to microservices. This approach:

1. Starts with a simpler deployment footprint
2. Enables quick development and iteration
3. Provides a clear path for future scalability
4. Maintains consistent performance and reliability

## Deployment Options

### Option 1: Docker-based Deployment

#### Prerequisites

- Docker v20+
- Docker Compose v2+
- Access to Docker Registry
- SSH access to target servers

#### Steps

1. **Build Docker Images:**
   ```bash
   # Build application image
   docker build -t portfolio-backend:latest .
   
   # Tag image for registry
   docker tag portfolio-backend:latest registry.example.com/portfolio-backend:$VERSION
   
   # Push to registry
   docker push registry.example.com/portfolio-backend:$VERSION
   ```

2. **Prepare Environment Configuration:**
   ```bash
   # Create .env file from template
   cp .env.production.example .env.production
   
   # Edit with appropriate values
   nano .env.production
   ```

3. **Deploy using Docker Compose:**
   ```bash
   # Copy docker-compose.yml and .env.production to server
   scp docker-compose.yml .env.production user@server:/opt/portfolio-backend/
   
   # SSH to server and deploy
   ssh user@server
   cd /opt/portfolio-backend
   docker-compose pull
   docker-compose up -d
   ```

4. **Verify Deployment:**
   ```bash
   # Check container status
   docker-compose ps
   
   # Check logs
   docker-compose logs -f
   
   # Test API
   curl http://localhost:3000/api/v1/health
   ```

### Option 2: Kubernetes Deployment

#### Prerequisites

- Kubernetes cluster (v1.22+)
- kubectl configured
- Helm (v3+)
- Access to container registry

#### Steps

1. **Build and Push Docker Image:**
   ```bash
   # Build and push as in Docker deployment
   docker build -t registry.example.com/portfolio-backend:$VERSION .
   docker push registry.example.com/portfolio-backend:$VERSION
   ```

2. **Prepare Kubernetes Manifests:**
   ```bash
   # Update image version in deployment.yaml
   sed -i "s/IMAGE_VERSION/$VERSION/g" kubernetes/deployment.yaml
   
   # Update ConfigMap with environment values
   kubectl create configmap portfolio-config --from-env-file=.env.production -n portfolio --dry-run=client -o yaml > kubernetes/configmap.yaml
   ```

3. **Deploy to Kubernetes:**
   ```bash
   # Apply database resources first
   kubectl apply -f kubernetes/database/
   
   # Wait for database to be ready
   kubectl wait --for=condition=ready pod -l app=portfolio-db -n portfolio --timeout=300s
   
   # Apply remaining resources
   kubectl apply -f kubernetes/
   ```

4. **Verify Deployment:**
   ```bash
   # Check pod status
   kubectl get pods -n portfolio
   
   # Check service
   kubectl get svc -n portfolio
   
   # Get API URL
   echo "API URL: http://$(kubectl get svc portfolio-backend -n portfolio -o jsonpath='{.status.loadBalancer.ingress[0].ip}')/api/v1/health"
   ```

### Option 3: Serverless Deployment (AWS)

#### Prerequisites

- AWS account with appropriate permissions

## Database Deployment

### PostgreSQL Setup

#### Self-Hosted Deployment

1. **Install PostgreSQL:**
   ```bash
   # Ubuntu/Debian
   apt-get update && apt-get install -y postgresql-13
   
   # Configure PostgreSQL for external connections
   nano /etc/postgresql/13/main/postgresql.conf
   # Update listen_addresses = '*'
   
   nano /etc/postgresql/13/main/pg_hba.conf
   # Add appropriate client authentication rules
   ```

2. **Create Database and User:**
   ```bash
   # Switch to postgres user
   sudo -u postgres psql
   
   # Create database and user
   CREATE DATABASE portfolio;
   CREATE USER portfolio_user WITH ENCRYPTED PASSWORD 'secure_password';
   GRANT ALL PRIVILEGES ON DATABASE portfolio TO portfolio_user;
   
   # Exit psql
   \q
   ```

3. **Run Migrations:**
   ```bash
   # Using npm script
   NODE_ENV=production npm run migrate
   
   # Or using database migration tool directly
   NODE_ENV=production npx sequelize-cli db:migrate
   ```

#### Cloud Managed PostgreSQL (AWS RDS)

1. **Create RDS Instance:**
   ```bash
   # Using AWS CLI
   aws rds create-db-instance \
     --db-instance-identifier portfolio-db \
     --db-instance-class db.t3.small \
     --engine postgres \
     --engine-version 13.7 \
     --allocated-storage 20 \
     --master-username portfolio_admin \
     --master-user-password "SecurePassword123!" \
     --db-name portfolio \
     --vpc-security-group-ids sg-12345678 \
     --db-subnet-group-name portfolio-subnet-group \
     --backup-retention-period 7
   ```

2. **Configure Security Groups:**
   ```bash
   # Allow application servers to connect to RDS
   aws ec2 authorize-security-group-ingress \
     --group-id sg-12345678 \
     --protocol tcp \
     --port 5432 \
     --source-group sg-87654321
   ```

3. **Run Migrations:**
   ```bash
   # Set DATABASE_URL environment variable
   export DATABASE_URL="postgresql://portfolio_admin:SecurePassword123!@portfolio-db.cxyzijk.us-east-1.rds.amazonaws.com:5432/portfolio"
   
   # Run migrations
   NODE_ENV=production npm run migrate
   ```

## CI/CD Pipeline Setup

### GitHub Actions Pipeline

Create `.github/workflows/deploy.yml`:

```yaml
name: Deploy Portfolio Backend

on:
  push:
    branches: [main, staging]
  workflow_dispatch:

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Set up Node.js
        uses: actions/setup-node@v3
        with:
          node-version: '16'
          cache: 'npm'
      - name: Install dependencies
        run: npm ci
      - name: Run tests
        run: npm test

  build:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v2
      - name: Login to Container Registry
        uses: docker/login-action@v2
        with:
          registry: registry.example.com
          username: ${{ secrets.REGISTRY_USERNAME }}
          password: ${{ secrets.REGISTRY_PASSWORD }}
      - name: Extract branch name
        shell: bash
        run: echo "BRANCH_NAME=$(echo ${GITHUB_REF#refs/heads/})" >> $GITHUB_ENV
      - name: Build and push
        uses: docker/build-push-action@v3
        with:
          context: .
          push: true
          tags: registry.example.com/portfolio-backend:${{ env.BRANCH_NAME }}-${{ github.sha }}

  deploy-staging:
    if: github.ref == 'refs/heads/staging'
    needs: build
    runs-on: ubuntu-latest
    environment: staging
    steps:
      - uses: actions/checkout@v3
      - name: Install kubectl
        uses: azure/setup-kubectl@v3
      - name: Set Kubernetes context
        uses: azure/k8s-set-context@v3
        with:
          kubeconfig: ${{ secrets.KUBECONFIG_STAGING }}
      - name: Deploy to staging
        run: |
          kubectl set image deployment/portfolio-backend portfolio-backend=registry.example.com/portfolio-backend:staging-${{ github.sha }} -n portfolio-staging
          kubectl rollout status deployment/portfolio-backend -n portfolio-staging

  deploy-production:
    if: github.ref == 'refs/heads/main'
    needs: build
    runs-on: ubuntu-latest
    environment: production
    steps:
      - uses: actions/checkout@v3
      - name: Install kubectl
        uses: azure/setup-kubectl@v3
      - name: Set Kubernetes context
        uses: azure/k8s-set-context@v3
        with:
          kubeconfig: ${{ secrets.KUBECONFIG_PRODUCTION }}
      - name: Deploy to production
        run: |
          kubectl set image deployment/portfolio-backend portfolio-backend=registry.example.com/portfolio-backend:main-${{ github.sha }} -n portfolio-production
          kubectl rollout status deployment/portfolio-backend -n portfolio-production
```

## Deployment Script

Create a deployment script `deploy.sh`:

```bash
#!/bin/bash
# Portfolio Backend Deployment Script

set -e

# Configuration
APP_NAME="portfolio-backend"
ENV=$1
VERSION=$(git rev-parse --short HEAD)
REGISTRY="registry.example.com"

# Validate environment
if [[ ! "$ENV" =~ ^(development|staging|production)$ ]]; then
  echo "Error: Environment must be 'development', 'staging', or 'production'"
  echo "Usage: $0 <environment>"
  exit 1
fi

# Load environment variables
echo "Loading $ENV environment variables..."
set -a
source .env.$ENV
set +a

# Build Docker image
echo "Building Docker image..."
docker build -t $APP_NAME:$VERSION .
docker tag $APP_NAME:$VERSION $REGISTRY/$APP_NAME:$ENV-$VERSION

# Push to registry
echo "Pushing to container registry..."
docker push $REGISTRY/$APP_NAME:$ENV-$VERSION

# Deploy based on environment
case $ENV in
  development)
    echo "Deploying to development environment..."
    docker-compose -f docker-compose.yml -f docker-compose.dev.yml up -d
    ;;
  staging)
    echo "Deploying to staging environment..."
    kubectl config use-context staging-cluster
    kubectl set image deployment/$APP_NAME $APP_NAME=$REGISTRY/$APP_NAME:$ENV-$VERSION -n portfolio-$ENV
    kubectl rollout status deployment/$APP_NAME -n portfolio-$ENV
    ;;
  production)
    echo "Deploying to production environment..."
    # Prompt for confirmation
    read -p "Are you sure you want to deploy to production? (y/n) " -n 1 -r
    echo
    if [[ ! $REPLY =~ ^[Yy]$ ]]; then
      echo "Deployment cancelled."
      exit 0
    fi
    
    kubectl config use-context production-cluster
    kubectl set image deployment/$APP_NAME $APP_NAME=$REGISTRY/$APP_NAME:$ENV-$VERSION -n portfolio-$ENV
    kubectl rollout status deployment/$APP_NAME -n portfolio-$ENV
    ;;
esac

echo "Deployment completed successfully!"
```

Make the script executable:

```bash
chmod +x deploy.sh
```

## SSL/TLS Configuration

### Using Let's Encrypt with Nginx

1. **Install Certbot:**
   ```bash
   apt-get update
   apt-get install -y certbot python3-certbot-nginx
   ```

2. **Obtain SSL Certificate:**
   ```bash
   certbot --nginx -d api.yourdomain.com
   ```

3. **Configure Nginx:**
   ```nginx
   server {
     listen 443 ssl;
     server_name api.yourdomain.com;
     
     ssl_certificate /etc/letsencrypt/live/api.yourdomain.com/fullchain.pem;
     ssl_certificate_key /etc/letsencrypt/live/api.yourdomain.com/privkey.pem;
     ssl_protocols TLSv1.2 TLSv1.3;
     ssl_prefer_server_ciphers on;
     ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:DHE-RSA-AES128-GCM-SHA256:DHE-RSA-AES256-GCM-SHA384;
     
     # Security headers
     add_header X-Content-Type-Options "nosniff" always;
     add_header X-Frame-Options "DENY" always;
     add_header X-XSS-Protection "1; mode=block" always;
     add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
     add_header Content-Security-Policy "default-src 'self'" always;
     add_header Referrer-Policy "strict-origin-when-cross-origin" always;
     add_header Permissions-Policy "geolocation=(), microphone=(), camera=()" always;
     
     location / {
       proxy_pass http://localhost:3000;
       proxy_http_version 1.1;
       proxy_set_header Upgrade $http_upgrade;
       proxy_set_header Connection 'upgrade';
       proxy_set_header Host $host;
       proxy_set_header X-Real-IP $remote_addr;
       proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
       proxy_set_header X-Forwarded-Proto $scheme;
       proxy_cache_bypass $http_upgrade;
     }
   }
   
   # Redirect HTTP to HTTPS
   server {
     listen 80;
     server_name api.yourdomain.com;
     return 301 https://$host$request_uri;
   }
   ```

4. **Set Up Auto-Renewal:**

## Database Backup and Restore

### Backup Script

Create a database backup script `backup-db.sh`:

```bash
#!/bin/bash
# PostgreSQL Backup Script

# Configuration
DB_NAME="portfolio"
DB_USER="portfolio_user"
BACKUP_DIR="/var/backups/portfolio"
DATE=$(date +%Y-%m-%d_%H-%M-%S)
BACKUP_FILE="$BACKUP_DIR/portfolio_$DATE.sql.gz"
RETENTION_DAYS=7

# Create backup directory if it doesn't exist
mkdir -p $BACKUP_DIR

# Perform backup
echo "Creating database backup: $BACKUP_FILE"
pg_dump -U $DB_USER $DB_NAME | gzip > $BACKUP_FILE

# Check if backup was successful
if [ $? -eq 0 ]; then
  echo "Backup completed successfully"
  
  # Upload to S3 if configured
  if [ ! -z "$S3_BUCKET" ]; then
    echo "Uploading backup to S3: $S3_BUCKET"
    aws s3 cp $BACKUP_FILE s3://$S3_BUCKET/backups/$(basename $BACKUP_FILE)
  fi
  
  # Clean up old backups
  echo "Cleaning up backups older than $RETENTION_DAYS days"
  find $BACKUP_DIR -name "portfolio_*.sql.gz" -mtime +$RETENTION_DAYS -delete
else
  echo "Backup failed!"
  exit 1
fi
```

### Restore Script

Create a database restore script `restore-db.sh`:

```bash
#!/bin/bash
# PostgreSQL Restore Script

# Configuration
DB_NAME="portfolio"
DB_USER="portfolio_user"
BACKUP_FILE=$1

# Check if backup file was provided
if [ -z "$BACKUP_FILE" ]; then
  echo "Error: No backup file specified"
  echo "Usage: $0 <backup-file>"
  exit 1
fi

# Check if backup file exists
if [ ! -f "$BACKUP_FILE" ]; then
  echo "Error: Backup file not found: $BACKUP_FILE"
  exit 1
fi

# Confirm restore operation
echo "WARNING: This will overwrite the existing database!"
read -p "Are you sure you want to restore from $BACKUP_FILE? (y/n) " -n 1 -r
echo
if [[ ! $REPLY =~ ^[Yy]$ ]]; then
  echo "Restore cancelled."
  exit 0
fi

# Perform restore
echo "Restoring database from backup: $BACKUP_FILE"
gzip -dc $BACKUP_FILE | psql -U $DB_USER $DB_NAME

# Check if restore was successful
if [ $? -eq 0 ]; then
  echo "Restore completed successfully"
else
  echo "Restore failed!"
  exit 1
fi
```

   ```javascript
   // src/shared/monitoring/prometheus.js
   const promClient = require('prom-client');
   const express = require('express');
   
   const register = new promClient.Registry();
   promClient.collectDefaultMetrics({ register });
   
   // Custom metrics
   const httpRequestsTotal = new promClient.Counter({
     name: 'http_requests_total',
     help: 'Total number of HTTP requests',
     labelNames: ['method', 'path', 'status'],
     registers: [register],
   });
   
   const httpRequestDuration = new promClient.Histogram({
     name: 'http_request_duration_seconds',
     help: 'HTTP request duration in seconds',
     labelNames: ['method', 'path', 'status'],
     buckets: [0.1, 0.3, 0.5, 0.7, 1, 3, 5, 7, 10],
     registers: [register],
   });
   
   // Middleware to track metrics
   const metricsMiddleware = (req, res, next) => {
     const start = Date.now();
     
     res.on('finish', () => {
       const duration = (Date.now() - start) / 1000;
       const path = req.route ? req.route.path : req.path;
       
       httpRequestsTotal.inc({
         method: req.method,
         path,
         status: res.statusCode,
       });
       
       httpRequestDuration.observe(
         {
           method: req.method,
           path,
           status: res.statusCode,
         },
         duration
       );
     });
     
     next();
   };
   
   // Metrics endpoint
   const metricsRoute = async (req, res) => {
     res.set('Content-Type', register.contentType);
     res.end(await register.metrics());
   };
   
   module.exports = {
     register,
     metricsMiddleware,
     metricsRoute,
   };
   ```

## Troubleshooting Deployment Issues

### Common Issues and Solutions

1. **Database Connection Issues:**
   - Check connection string in environment variables
   - Verify network connectivity between application and database
   - Ensure database credentials are correct
   - Check database security group/firewall rules

2. **Redis Connection Issues:**
   - Verify Redis connection string
   - Check Redis server status
   - Ensure Redis credentials are correct
   - Check network connectivity

3. **Application Not Starting:**
   - Check logs for startup errors
   - Verify all required environment variables are set
   - Check file permissions
   - Verify disk space is available

4. **SSL/TLS Issues:**
   - Check certificate validity and expiration
   - Verify certificate and private key match
   - Ensure certificate chain is complete
   - Check SSL/TLS configuration

## Rollback Procedures

### Docker-based Rollback

```bash
# List available images
docker images | grep portfolio-backend

# Roll back to previous version
docker-compose down
export IMAGE_TAG=previous-version
docker-compose up -d
```

### Kubernetes Rollback

```bash
# List deployment history
kubectl rollout history deployment/portfolio-backend -n portfolio

# Roll back to previous revision
kubectl rollout undo deployment/portfolio-backend -n portfolio

# Roll back to specific revision
kubectl rollout undo deployment/portfolio-backend -n portfolio --to-revision=2

# Check rollback status
kubectl rollout status deployment/portfolio-backend -n portfolio
```

### Database Rollback

```bash
# Restore from backup
./restore-db.sh /var/backups/portfolio/portfolio_2023-03-21_00-00-00.sql.gz

# Run down migrations
NODE_ENV=production npm run migrate:down

# Run down to specific migration
NODE_ENV=production npx sequelize-cli db:migrate:undo:all --to YYYYMMDDHHMMSS-migration-name.js
```
